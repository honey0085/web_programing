# 냉정한 평가: 프로젝트 리더(프롬프터)로서의 역량 진단

작성 목적: 본 문서는 현재 진행 중인 '반응속도 테스트' 프로젝트에서의 당신(프로젝트 리더·프롬프터)을 객관적이고 냉정하게 평가하고, 개선 가능한 구체적 행동 지침과 추적 가능한 KPI를 제시하기 위해 작성되었습니다.

---

## 1) 평가 대상과 전제
- 평가 대상: 프로젝트 리더이자 프롬프터(요구 정의 및 AI 지시 역할)
- 전제: 소규모 웹 프로토타입(단일 HTML 파일 기반), 빠른 반복과 AI 협업을 통해 기능 요구를 반영해 온 상태
- 평가 관점: 리더십(의사결정·우선순위), 프롬프터 능력(프롬프트 설계·반복), 기술 역량(코드·테스트), 협업·커뮤니케이션, 프로세스·생산성

---

## 2) 한줄 요약(냉정하게)
- 강점: 빠르게 요구를 명확화하고 실무 결과물을 빠르게 도출해내는 실행력은 우수하다.
- 개선 필요: 장기적 관점의 설계·검증, 프롬프트의 재현성·문서화, 팀 공유와 책임 분담에서 구조화가 부족하다.

---

## 3) 상세 평가

### A. 리더십 (의사결정 · 우선순위)
- 잘하는 점
  - 빠른 의사결정으로 프로토타입을 신속하게 전개함.
  - 실험 우선순위를 명확히 해 최소 기능(MVP)을 빠르게 구현.
- 개선점
  - 의사결정 근거(왜 이 옵션을 택했는지) 문서화가 부족하여 후속 검증이 어렵다.
  - 리스크(측정 정확도, 접근성 등)에 대한 사전 고려와 완화 계획이 약함.

### B. 프롬프터 능력 (프롬프트 설계 · 반복)
- 잘하는 점
  - 요구사항을 명확히 제시하면 즉시 코드 반영 및 개선을 이끌어냄.
  - 빠른 피드백 루프를 통해 기능을 빠르게 완성함.
- 개선점
  - 프롬프트의 재현성(버전 관리)이 없음 — 동일한 결과를 재생성하기 어려움.
  - 프롬프트 입력에 대한 기대 출력(테스트 케이스)이 명확하지 않아 결과 검증이 수동으로 이루어짐.

### C. 기술 역량 (코드 품질 · 검토)
- 잘하는 점
  - 기능 구현 요구를 기술적으로 간결하게 전달함.
- 개선점
  - 원인 기반 문제 해결보다 증상 수정에 머무르는 경향(예: 임시 타이머 취소 누락 보완 등).
  - 테스트(단위/통합)의 적용이 부족하여 회귀 위험 존재.

### D. 커뮤니케이션 & 협업
- 잘하는 점
  - 목표와 데모 결과로 팀/스테이크홀더를 설득하는 데 강함.
- 개선점
  - 역할 위임과 책임 흐름이 불분명 — 누가 무엇을 확인/병합/배포하는지 명확하지 않음.
  - 문서·회의록 정리가 부족해 반복 질문과 반복 작업이 발생함.

### E. 프로세스 및 생산성
- 잘하는 점
  - 빠른 실험 사이클(요구→수정→재검증)을 잘 운영함.
- 개선점
  - 우선순위 기준(가치·리스크·노동량)에 대한 체크리스트가 없음.
  - 장기 개선 과제(측정 정밀도, 데이터 저장, 분석)보다 단기 기능 완성에 치중함.

---

## 4) 핵심 약점(우선순위 기준)
1. 프롬프트/요구의 재현성 미흡(재사용 가능한 형식 미비)
2. 문서화/이슈 트래킹 부족(결정 근거·테스트 케이스 부재)
3. 테스트 자동화 부재로 회귀 발생 위험
4. 역할·책임의 비구조화(개발·배포 프로세스의 소유권 불명확)

---

## 5) 구체적 개선 권장사항(실무형, 우선순위 포함)

### 단기(1~2주)
- 프롬프트 템플릿 도입
  - 목적, 입력(컨텍스트), 기대 출력, 검증 방법(예시)을 포함한 템플릿을 만들고 모든 프롬프트에 적용.
  - 예: "기능: 3회 평균 측정. 입력: index.html 원본. 출력: 수정된 index.html. 검증: 3회 측정 동작 확인 스크린샷/로그".
- 결정 기록(Decision Log) 작성
  - 주요 의사결정 1페이지 요약(왜 선택했는지, 대안, 리스크, 다음 액션)으로 남김.
- 최소한의 테스트 수동 체크리스트 작성
  - 성공/실패 케이스(조급 클릭, 정상 클릭, 브라우저 재시작 등)를 문서화.

### 중기(2~6주)
- 프롬프트 버전관리 및 결과 재현성 확보
  - 프롬프트는 Git에 텍스트 파일로 저장(예: `prompts/` 폴더), 각 변경에 이유를 코멘트로 첨부.
- 테스트 자동화 시작
  - 간단한 E2E 스크립트(Playwright 또는 Puppeteer)로 ‘3회 측정’ 시나리오 자동화.
- 코드 분리 및 모듈화
  - `index.html`의 JS를 분리하여 `app.js`로 관리하고 간단한 빌드/호스팅 방식 마련.

### 장기(2~6개월)
- 데이터 수집 및 분석 파이프라인 구축
  - 세션 데이터를 CSV/서버로 저장, 대시보드(Chart.js 등)로 추세 시각화.
- 팀 내 역할 정의와 배포 파이프라인 구축
  - PR 리뷰 규칙, 병합 권한, 릴리스 체크리스트 수립.
- 개인 역량 강화
  - 테스트 주도 개발(TDD) 기본 과정 수료, 프롬프트 설계 심화 학습.

---

## 6) 측정 가능한 KPI(권장)
- 프롬프트 재현성 지표: 동일 프롬프트로 1회 이상 동일 출력 재생성 비율 (목표 90%)
- 테스트 커버리지(기능별 E2E 시나리오 수): 초기에 0 → 3개(단기) → 10개(중기)
- 문서화 비율: 중요한 결정 100%에 대해 Decision Log 작성 비율 (목표 100%)
- 회귀 발생률: 릴리스 후 버그 리포트 비율(목표 감소)

---

## 7) 프롬프트 체크리스트(실전용)
- 목적이 명확한가? (무엇을 얻고 싶은가)
- 입력 컨텍스트는 충분한가? (파일, 환경 설명 포함)
- 기대 출력은 검증 가능한가? (예: 코드, 테스트 케이스, 수치)
- 실패·예외에 대한 처리 지침이 있는가? (예: 잘못된 입력 시 반환값)
- 변경 기록을 남겼는가? (버전·업데이트 이유)

---

## 8) 30/60/90일 개인 액션 플랜(추천)
- 30일: 프롬프트 템플릿 도입, Decision Log 시작, 3개 수동 테스트 문서화
- 60일: 간단한 E2E 스크립트(3개 시나리오) 작성, 프롬프트 버전관리 정착
- 90일: 데이터 저장/시각화 PoC, 팀 배포 프로세스 초안 완성

---

## 9) 마지막 한마디(직설적 평가)
당신은 빠르게 결과물을 만들어내는 실행자다. 그러나 빠름이 곧 성공을 보장하지는 않는다. 동일한 속도로 '재현 가능하고 검증 가능한 결과'를 만들도록 프로세스와 문서화를 습관화하지 않으면, 시간이 지날수록 기술 부채와 비효율이 누적될 것이다. 지금의 실행력을 유지하되, '재현성·검증·권한 분배'에 투자하라. 단기적 불편(문서·테스트 작성)은 중기적 생산성 폭발을 불러온다.
