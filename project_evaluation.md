## 주요 문제(우선순위 기준)
1. 측정 정확도 문제 (우선순위: 높음)
   - `Date.now()` 기반 시간 측정은 환경에 따라 미세한 오차가 발생할 수 있음. 고해상도 타이머(`performance.now()`) 권장.
2. 코드 구조 및 분리 부족 (우선순위: 중상)
   - JS가 HTML에 인라인으로 포함되어 있어 재사용, 테스트, 정적 분석이 어려움.
3. 테스트 자동화 부재 (우선순위: 중상)
   - 단위·E2E 테스트가 없어 회귀 위험 존재.
4. 보안 및 배포 관행 미비 (우선순위: 중)
   - 인라인 스크립트로 인해 CSP 적용이 어렵고, 장기 배포 시 보안 설정 필요.
5. 접근성 고려 부족 (우선순위: 중)
   - ARIA, 스크린리더 호환성, 키보드 네비게이션 보완 권장.

## 기술적 권장 개선 (단기/중기/장기)

### 단기(0–2일)
- `Date.now()` → `performance.now()`로 교체하여 측정 정밀도 개선.
- JS 분리: `app.js`로 로직 분리 후 `index.html`에서는 로드만 수행.
- 입력 validation: `bestRecord` 저장/로딩 시 타입 및 범위 검증 추가.

### 중기(1–3주)
- E2E 테스트 도입(Playwright 또는 Puppeteer): 조급 클릭, 정상 흐름(3회) 등 3개 시나리오 자동화.
- ESLint/Prettier 적용 및 CI 연동으로 코드 품질 유지.
- 접근성 향상: ARIA 속성 추가, 키보드 안내 문구 개선.

### 장기(1–3개월)
- 데이터 수집 및 시각화: 세션 기록을 서버 또는 CSV로 저장하고 Chart.js로 시각화.
- 배포 파이프라인 구성: 릴리스 체크리스트, Decision Log, PR 리뷰 규칙 수립.

## 안전 및 품질 체크리스트(권장)
- 모든 타이머에서 `clearTimeout`이 호출되고 상태 초기화가 확실히 이루어지는가?
- `performance.now()` 적용 후 측정 오차가 허용범위 내에 있는가? (목표: 평균 오차 < 5%)
- localStorage 키(`bestRecord`) 파싱 시 NaN 방지 및 자료형 검증이 되는가?
- JS 분리 후 ESLint 경고/오류가 없는가?
- E2E 시나리오(3개) 자동화로 성공률이 100%에 가까운가?
